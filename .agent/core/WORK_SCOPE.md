# WORK_SCOPE

## 1. Як фіксується scope

Scope задачі складається тільки з:

- явного запиту користувача;
- прямих обмежень у запиті;
- мінімуму змін, без яких задачу неможливо завершити.

## 2. Що є виходом за межі scope

Поза scope є:

- зміни, яких користувач не просив;
- побічні виправлення;
- рефакторинг без окремого запиту;
- перейменування;
- зміни архітектури;
- зміни конфігурацій, інфраструктури, CI/CD, залежностей, prod-конфігів;
- нові migrations (NNN_*.sql) — поза scope, якщо тільки задача
  не класифікована як FEATURE MINOR або MAJOR і план підтверджений
  (див. AGENTS.md розділ 5.6 і 7.3);
- додаткові документи без прямої потреби;
- поради, плани або `helpful extras` без прямого запиту.

## 3. Що заборонено без прямого дозволу

Заборонено без прямого дозволу:

- змінювати файли, не потрібні для задачі;
- включати несуміжні файли в аналіз або звіт без потреби;
- виправляти знайдені попутно дефекти;
- розширювати задачу з власної ініціативи;
- робити висновок "це теж треба" як підставу для змін;
- вносити приховані побічні зміни.

## 4. Якщо знайдено побічну проблему

- Не виправляй її.
- Не включай її в поточний scope.
- Згадай її тільки в `Недоведено / ризики`, якщо вона прямо впливає на результат задачі.

## 5. Якщо межа задачі неочевидна

- Не домислюй scope самостійно.
- Зупинись і застосуй протокол ескалації (AGENTS.md розділ 11):
  явно сформулюй що невідомо, запропонуй варіанти A/B,
  не виконуй нічого до отримання відповіді.
- Якщо навіть варіанти неможливо сформулювати — постав задачу
  в статус Incomplete і заповни поле Недоведено / ризики.
